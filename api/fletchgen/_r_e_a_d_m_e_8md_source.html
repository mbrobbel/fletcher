1 # Fletchgen: The Fletcher Design Generator
2 
3 Fletchgen is a command-line utility that generates the upper layers of a
4 hardware design, including simulation and platform-specific top-levels, based on
5 Arrow Schemas and Recordbatches.
6 
7 #### What does Fletchgen generate?
8 
9 - For each **Arrow Field**, an **ArrayReader/Writer** is instantiated.
10 - For **each Arrow Schema**, a **RecordBatchReader/Writer** is generated and
11  wrapper around all ArrayR/W instances.
12 - For **all Arrow Schemas**, a **Kernel template** is generated.
13 - The Kernel is wrapped in a **Nucleus** that hides low-level control flow to
14  RecordBatchR/W's and instantiates the
15  MMIO component that handles AXI4-lite memory-mapped input/output.
16 - All RecordBatchR/W's and the Nucleus is wrapped by a **Mantle**. The Mantle
17  also instantiate the required memory bus interconnection logic.
18 
19 This can also be shown schematically as follows:
20 ![Fletchgen output, schematically](./docs/fletchgen.svg)
21 
22 #### What does Fletchgen NOT generate?
23 
24 Fletchgen does **not** generate the kernel itself.
25 
26 #### What top-levels does Fletcher generate?
27 
28 It currently supports only two top-level platforms.
29 
30 - One platform is a **simulation top-level** that uses a memory model that can
31  be filled with RecordBatches.
32  - To enable this top-level, use the `--sim` flag.
33 - The other is an **AXI top-level** that has an AXI4 (full) master port and
34  AXI4-lite slave port.
35  - To enable this top-level, use the `--axi` flag.

43 # Build & install
44 
45 Fletchgen is under heavy development, so for now we'd recommend to make a debug
46 build using CMake as follows:

49 git clone https://github.com/abs-tudelft/fletcher.git --recursive
50 cd fletcher/codegen/cpp/fletchgen
51 mkdir debug
52 cd debug
53 cmake ..
54 make
55 sudo make install

72 # Supported/required metadata for Arrow Schemas
73 
74 Fletchgen derives how to use an Arrow Schema from attached key-value metadata
75 that is stored in Arrow Schemas. You can use this to, for example, prevent the
76 generation of hardware structures for fields that you're not going to use in
77 your kernel implementation.
78 
79 ## Schema metadata:
80 
81 | Key | Possible values | Default | Description |
82 | ----------------- | --------------- | ------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
83 | fletcher_name | any string | none | The name of the schema. This is required for the schema to be identifiable after hardware generation. |
84 | fletcher_mode | read / write | read | Determines whether a RecordBatch of this schema will be read or written by the kernel. |
85 | fletcher_bus_spec | aw,dw,lw,bs,bm | 64,512,8,1,16 | Key to set the bus specification of the RecordBatchReader/Writer resulting from this schema. aw: address width, dw: data width, lw: burst length width, bs: minimum burst size, bm: maximum burst size. |
86 
87 ## Field metadata:
88 
89 | Key | Possible values | Default | Description |
90 | ------------------ | --------------- | ------- | ------------------------------------------------------------------------------------------------------------------------------------- |
91 | fletcher_ignore | true / false | false | If set to true, ignore a specific schema field, preventing generation of hardware to read/write from/to it. |
92 | fletcher_epc | 1 / 2 / 4 / ... | 1 | Number of elements per cycle for this field. For `List<X>` fields where X is a fixed-width type, this applies to the `values` stream. |
93 | fletcher_lepc | 1 / 2 / 4 / ... | 1 | For `List<primitive>` fields only. Number of elements per cycle on the `length` stream. |
94 | fletcher_profile | true / false | false | If set to true, mark this field for profiling. The hardware streams resulting from this field will have a profiler attached to them. |
95 | fletcher_tag_width | 1 / 2 / 3 / ... | 1 | Width of the `tag` field of commands and unlock streams of RecordBatchReaders/Writers. Can be used to identify commands. |
